## 5. What Worked
> *What guidelines or practices clearly improved your Week 13 app compared to Week 2? Be specific. Point to concrete examples in your code or AI interactions.*


The following applied guidelines produced clear, observable improvements in the Week 13 app compared to Week 2.

- **Guideline 4.6 (Pre-Mortem Prompting)**: Prompting the LLM with concrete failure scenarios before writing any auth code surfaced the `userId` vs. `id` property mismatch and token race condition before they were ever introduced. In Week 2, the same class of problem was discovered only after edit buttons stopped appearing — costing some time of debugging. Week 13 had zero authorization bugs at submission.

- **Guideline 4.3 (Assign Role/Persona to LLM)**: Asking the LLM to act as a "senior software architect" during requirements generation produced comprehensive edge-case handling and consistent middleware chain design. In Week 2 with no role, the LLM produced minimal auth setting with no rate limiting, no helmet headers, and a hardcoded secret key.

- **Guideline 6.1 (Work in Short Iterative Cycles)**: Breaking development into 6 explicit cycles (Auth → Posts → CRUD → Replies → Profiles → UI) meant each feature was tested before the next began. In Week 2, features were built in large batches; bugs accumulated and were only discovered during integration, which require huge amount of debugging sessions.

- **Guideline 6.4 (Specify Project-Specific Tools & Workflows)**: Declaring the full stack (Express, JWT, bcrypt, Zustand, Axios interceptors) upfront eliminated LLM suggestions for incompatible libraries. All 10 backend packages installed and ran without modification. In Week 2, three dependency conflicts required manual identification and fixed takes good amount of time.

- **Guideline 6.6 (Specify Input/Output Formats)**: Defining JSON request/response schemas for every endpoint before generating frontend code that produced zero API mismatch errors. In Week 2, the frontend and backend were built independently, leading to format mismatches discovered during integration.

- **Guideline 9.2 (Generate-Validate-Repair Loop)**: Treating each feature as a generate → curl-test → repair → retest cycle achieved a 100% pass rate (15/15 tests). In Week 2, features were declared done when they "looked right" without systematic testing; at least 3 testing related problem undiscovered during submission.

- **Guideline 8.1 (Respond-Collaborate-Followup Debugging)**: The 3-step debugging approach (identify symptom → investigate with logs → fix and verify) reduced per-bug resolution time from ~30 minutes (Week 2) to ~10 minutes (Week 13). The `userId` vs. `id` mismatch in `postController.js` was identified and fixed in one structured session rather than through trial-and-error.

- **Guideline 12.4 (Structured Logging with Standardized Formats)**: Establishing a consistent log format `{ timestamp, level, userId, action, details, result }` across all controllers made log output machine-readable and easy to filter. In Week 2, ad-hoc `print()` statements were scattered and had no consistent structure, making debugging from logs impractical.

## 6. What Did Not Work
> *What guidelines did not help, or even made things harder? Were there situations where following a guideline produced worse results? Be honest.*


The following reflects gaps from guidelines that did not work well in the app and one case where an applied guideline produced problematic returns.

- **Guideline 9.3 (Explicitly Request Boundary and Negative Cases) - not applied**: Testing only covered the main user flows and authorization checks. We did not request any boundary tests, such as posts with exactly 280 characters, empty strings, or very long usernames. Because of this, some edge cases were never tested, so there is a risk that issues could go unnoticed even if all 15 tests passed. 

- **Guideline 9.5 (Fault Model / Mutation Mindset) - not applied**: We judged test quality only by the pass rate, not by how well the tests covered possible faults. Without thinking about how to break the code, tests could pass while still missing entire groups of bugs, like off-by-one errors in reply nesting or race conditions when writing to JSON files. This is a known weakness in our test suite. 

- **Guideline 9.4 (Automated Testing Frameworks) - not applied**: Using curl for manual testing worked for 15 tests, but it did not scale. As the app grew, running all tests by hand after each change became tiring. Automated tests with Jest or Supertest would have caught regressions, but we chose not to use them to save time. This trade-off would not work for a longer project. 

- **Guideline 10.10 (Automated CI Gates) - not applied**: Since we did not have a CI pipeline, there was no automatic check to stop a broken commit from being added. We checked everything by hand before each commit. Twice, a manual test passed, but a problem was only found in the next development session. A simple GitHub Actions workflow would have caught this. 

- **Guideline 11.1 (Analyze Complexity Before Writing Final Code) - not applied**: Performance analysis was done after problems appeared. For example, we found the reply-filtering bottleneck (see Guideline 11.4) only after the feature was finished, not during design. If we had analyzed complexity earlier, we might have used an indexed lookup from the start and avoided extra work.

- **Guideline 12.3 (Avoid Logging on Hot Paths) - not applied**: We added structured logging (Guideline 12.4) to all controllers without checking which ones were used most. The post feed endpoint, which runs on every page load, logs a full structured object each time. At scale, this could slow things down due to extra I/O. The logging guideline made it easier to observe the system but also created a hidden performance issue.

- **Guideline 10.7 (Require Evidence-Grounded Justification) - applied, but diminishing returns**: Requiring proof before accepting AI optimization claims helped avoid unnecessary complexity, but it also slowed down some simple, obviously correct changes. In practice, using this rule for every suggestion, even small refactors, added friction without enough benefit. It works well for performance claims but is too much for cosmetic or structural changes.

## 7. Reflection
> *Looking back across both apps: How has your mental model of working with AI changed? What would you do differently if you built the app a third time?*

**How my mental model of working with AI changed**

Before Week 13, I saw LLM as just a tool for generating code. I would give it a feature description, take whatever it gave me, and then fix any problems myself. For example, I built the Week 2 app this way. My prompts were short and vague, like "build a Twitter-like app" or "add authentication." The LLM responded with code that worked, but it missed important security details, such as rate limiting, using a hardcoded secret key, and enabling wildcard CORS with credentials. I didn’t know how to ask for more because I didn’t have a clear idea of what was missing. 

Going through the guidelines changed my approach. LLM won’t point out issues like unsafe CORS settings or remind you to use environment variables for secrets unless you ask it the right way. Assigning a persona (Guideline 4.3), describing possible failures upfront (Guideline 4.6), and specifying the tool chain (Guideline 6.4) aren’t just extra steps. They help the LLM understand enough to give you production-ready code instead of just a prototype. With better prompts, the same AI model gave me much better results. This was a big realization for me: the AI’s limits were really about how I was prompting it. 

Another important lesson was that using LLM for development still needs a solid process. Short feedback cycles (Guideline 6.1), generate-validate-repair loops (Guideline 9.2), and structured debugging (Guideline 8.1) aren’t just borrowed from traditional software development. They are even more important when working with LLM, because mistakes from unchecked AI code can quickly spread through a project.

**What I would do differently for a third app**

1. **Apply guidelines from the first prompt, not retroactively.** The security issues in the Week 2 app, such as hardcoded secrets and wildcard CORS, could have been easily avoided by applying Guideline 4.6 during the requirements phase. Retrofitting security is always more costly than integrating it from the start. 

2. **Set up automated testing before writing features.** The lack of Guideline 9.4 (automated testing framework) required manual re-running of the 15 test suite after each change. Implementing Supertest or Jest at the scaffolding stage would have taken 30 minutes upfront and saved hours of manual testing. 

3. **Define a fault model before writing tests.** Achieving a 15/15 pass rate felt sufficient, but without Guideline 9.5 (mutation mindset), the test suite may not address the most critical failure modes. A third app should begin with a list of likely faults, such as off-by-one errors in pagination, concurrent write conflicts on JSON files, and malformed JWT payloads, and develop tests to target these issues.

4. **Profile before logging.** Guideline 12.4 (structured logging) improved observability but resulted in logging every request to the post feed endpoint. A third app should apply Guideline 12.3 (avoid logging on hot paths) alongside structured logging, treating both as interconnected concerns. 

5. **Add a CI gate from the first commit.**  Without Guideline 10.10 (automated CI gates), broken states entered the codebase at least twice and were only detected in subsequent sessions. A single GitHub Actions workflow running the test script on every push would have prevented this at minimal cost.
